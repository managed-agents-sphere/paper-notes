# Dive into Claude Code

## 1. TL;DR

源码级（TypeScript v2.1.88）逆向分析 Claude Code，提炼出 **5 价值观 → 13 设计原则 → 7 组件 / 5 层架构** 的映射，并与 OpenClaw（多通道个人助理 gateway）做架构对照。**核心结论**：reasoning（model）与 enforcement（harness）严格分离——社区估计 *只有 1.6% 的代码是 AI 决策逻辑，98.4% 都是 operational infrastructure*；而 infrastructure 的重头戏是把 **context window 当作唯一硬约束**——围绕它叠了一条 **5 层渐进式 compaction pipeline**（budget → snip → microcompact → collapse → auto-compact），外加 cache-aware 的特殊处理（实验数据：非 cache 复用路径有 98% cache miss、占舰队 cache_creation 成本的 ~0.76%）。

### 说人话

> Claude Code 真正聪明的地方**不在 LLM**，在 LLM 外面那层"运营基础设施"。这篇论文把这层基础设施一寸一寸拆开，告诉你**每个设计决策背后回答的是哪个工程问题**。对做 agent 系统的人来说，这是一份**抄作业指南**。

#### 术语翻译

| 原词 | 说人话 |
|---|---|
| **agentic loop** | 一个 while(true) 循环：让模型说一句 → 看它要不要调工具 → 调完把结果塞回去 → 再让模型说一句。直到模型不再要工具为止 |
| **harness** | 包在 LLM 外面的"骨架代码"——做权限、context 管理、工具分发、错误恢复，**LLM 自己碰不到文件系统/Shell** |
| **context window** | 一次塞给模型的 token 总额上限。Claude 4.6 是 1M，老型号 200K。**这是整个系统的硬约束** |
| **compaction** | "压缩对话历史"。当 context 快塞满时，把老的 messages 换成摘要 |
| **cache（prompt cache）** | Anthropic API 的 KV 缓存：**前缀相同的请求只算一次钱**（5 分钟 TTL）。前缀一变就全失效——所以"改不改前缀"是个钱的问题 |
| **read-time projection** | 不修改原数组，**临时 view 出一份压缩版**给模型看，原始历史还留着——和 SQL 视图一样 |
| **append-only JSONL** | 永远只往文件尾巴加行，不改不删——这样 resume / fork / rewind 都能精准重放 |
| **ReAct** | Reason + Act 循环范式（Yao 2022）。本文反复对比的也是 LangGraph（图状控制流）/ Devin（显式 plan）/ Aider（git rollback as safety） |

#### 关键直觉的类比

**5 层 compaction = 渐进式祛痘**：先用洁面（budget 限单条工具结果）→ 化妆水（snip 掐尾巴）→ 精华（microcompact 看 cache 删局部）→ 概念性遮瑕（collapse 不删原料、只 view 出干净版）→ 最后才动激光（auto-compact 整段重写）。**便宜的先来，贵的最后才用**。

**model vs harness = 大脑 vs 脊髓反射**：模型负责"想"，harness 负责"做"——而且**harness 会把模型想的危险动作直接拒掉**，不让动作穿透到执行环境。这是为什么"被 prompt injection 骗了的模型也越不过 deny-list"。

**context collapse = 数据库视图**：原始 history 一行不动还在 REPL 数组里，**collapse 只是改了"读出来时呈现成什么样"**。所以"压缩"完全可逆——这正是 resume / fork / rewind 能成立的前提。

#### "context 是唯一硬约束"什么意思

整个 Claude Code 的工程取舍都围绕 **不要让 context 提前用完**：CLAUDE.md 嵌套目录懒加载、ToolSearch 只挂工具名、subagent 只回 summary、单条 tool result 限长——**每一条都在抢 token 预算**。换句话说，做 agent 系统**先想清楚 context 怎么花，再谈别的**。

## 3. 问题与动机

agentic coding tool（Claude Code、Cursor、Devin 等）已从"补全式"演化为"自主多步执行"，**架构需求与 Copilot 完全不在一个维度**。但：

- Anthropic 只发**用户文档**，不公布架构细节
- 学界对 production agent 的 design space **缺一份系统化总结**
- 业内反复造轮子（LangGraph、SWE-Agent、OpenHands、Aider 各自一套），没人把"recurring design questions"显式列出来

作者用源码级方法论做了三件事：

1. **identify** 5 个 human values → 13 个 design principles → 一组 recurring design questions
2. **trace** 每个 principle 落到具体 source file（query.ts / permissions.ts / context.ts / compact.ts ...）
3. **contrast** 与 OpenClaw（gateway 范式）对比，证明同一组问题在不同 deployment context 下答案不同

> "The model's only interface to the outside world is the structured tool_use protocol, which the harness validates before execution."

## 4. 方法

不是 ML 方法论，是**架构剖析方法论**：

- 主证据：可获取的 TypeScript 源码（v2.1.88，~Apr 2026）
- 次证据：Anthropic 官方文档 + 社区分析（Cherny & Wu 2025、Hughes 2026 等）
- 数字旁证：Anthropic 132 名工程师内部调研（Huang et al. 2025）

### 4.1 5 个 Human Values（架构动机）

1. **Human Decision Authority** — 人保留最终决策权（principal hierarchy: Anthropic > operators > users）
2. **Safety, Security, Privacy** — 系统义务，**与"人是否上心"解耦**
3. **Reliable Execution** — 单轮正确 + 跨 context 边界连贯
4. **Capability Amplification** — Anthropic 调研：~27% 任务"没这工具不会做"
5. **Contextual Adaptability** — 信任随时间共构（auto-approve 率从 <50 sessions 的 ~20% → 750 sessions 的 >40%）

### 4.2 13 Design Principles（与 context 工程相关的几条加粗）

| 原则 | 服务的 value | 设计问题 |
|---|---|---|
| Deny-first with human escalation | Authority, Safety | 未识别动作允许/拒绝/上报？ |
| Graduated trust spectrum | Authority, Adaptability | 固定权限 or 可演化？ |
| Defense in depth | Safety, Authority, Reliability | 单层 or 多层叠加？ |
| Externalized programmable policy | Safety, Authority, Adaptability | 硬编码 or 配置化 + lifecycle hooks？ |
| **Context as scarce resource with progressive management** | Reliability, Capability | **唯一资源约束，单段截断 or 渐进 pipeline？** |
| Append-only durable state | Reliability, Authority | 可变状态 or 追加日志？ |
| Minimal scaffolding, maximal operational harness | Capability, Reliability | 投资 scaffolding-side reasoning or operational infra？ |
| Values over rules | Capability, Authority | 死规则 or 上下文判断 + 兜底闸？ |
| Composable multi-mechanism extensibility | Capability, Adaptability | 单 API or 分层（不同 context cost）？ |
| Reversibility-weighted risk assessment | Capability, Safety | 一刀切 or 可逆/只读放宽？ |
| Transparent file-based config and memory | Adaptability, Authority | 不透明 DB / embedding or 用户可见可版本控制的文件？ |
| Isolated subagent boundaries | Reliability, Safety, Capability | 共享父 context or 隔离？ |
| Graceful recovery and resilience | Reliability, Capability | 硬失败 or 静默恢复 + 必要时上报？ |

## 5. 实验

不是 ML 实验，是**架构对照实验**——和 **OpenClaw**（多通道个人助理 gateway，2026）做六维对比，证明"deployment context 不同 → 同一问题的答案不同"。

| 维度 | Claude Code | OpenClaw |
|---|---|---|
| System scope | CLI/IDE，per-session ephemeral process | 持久 daemon（loopback WS gateway） |
| Trust model | 不信任模型，per-action 权限 + ML 分类器 + 7 模式 | 信任单 operator，perimeter 级 ACL + DM pairing |
| Agent runtime | `queryLoop()` 即系统中心 | embedded Pi-agent，被 gateway 控制平面包住 |
| Extension | 4 机制（MCP/plugin/skill/hook）按 **context cost** 排序 | manifest-first 12 capability 类型，注册到中央 registry |
| Memory & context | CLAUDE.md 4 级 + 5 层 compaction + LLM scan 选 memory file | AGENTS.md/SOUL.md/MEMORY.md 多文件 + 可插拔 compactor + 可选向量混合检索 + 实验性 "dreaming" 长程提升 |
| Multi-agent | task-delegating subagent + worktree 隔离 + 只回 summary | 双轨：(a) 跨 agent routing；(b) sub-agent 嵌套（max 5，默认 1，建议 2） |

## 6. 关键结果

> ⚠️ 这篇是**架构论文**，"结果"指的是**结构性发现**，不是数字。但有几个关键数字值得记。

### 6.1 Context 工程：5 层 compaction pipeline（最值得抄）

每次 model call 前**顺序**执行，便宜的先：

| # | Shaper | 触发与作用 | 关键工程细节 |
|---|---|---|---|
| 1 | **Budget reduction**（always on） | 单条 tool result 超 `maxResultSizeChars` 就替换成 content reference | 替换记录持久化以便 resume 重建；仅按 tool_use_id 工作 |
| 2 | **Snip**（HISTORY_SNIP gated） | 轻量删老 history | snipTokensFreed **必须显式上报给 auto-compact** —— token 计数器只看最近一条 assistant message 的 usage 字段，snip 的省量否则不可见 |
| 3 | **Microcompact**（CACHED_MICROCOMPACT） | 细粒度压缩 | **boundary 延后到 API 响应回来后再算**，用 *实测* `cache_deleted_input_tokens` 而非估值，返回 `pendingCacheEdits` |
| 4 | **Context collapse**（CONTEXT_COLLAPSE） | **read-time projection**，不动 REPL 历史 | summary 写到 collapse store，不进 REPL array —— 这是 collapse 能跨 turn 持久的原因 |
| 5 | **Auto-compact**（默认开） | 让模型生成全文摘要 | 前 4 步还不够才触发；PreCompact hooks 先跑；GrowthBook flag 控是否复用主对话的 prompt cache |

> **金句（值得贴在工位上）**：
> "no single compaction strategy addresses all types of context pressure. Budget reduction targets individual tool outputs that overflow size limits. Snip handles temporal depth. Microcompact reacts to cache overhead. Context collapse manages very long histories. Auto-compact performs semantic compression as a last resort."

### 6.2 Cache-aware 的硬数据

源码注释里有 January 2026 实验：
> "false path is 98% cache miss, costs ∼0.76% of fleet cache_creation"

也就是说：**compaction 时是否复用主对话的 prompt cache，会带来 ~0.76% 的舰队级 cache_creation 成本差**。这是把 "compaction 与 prompt cache 协同" 做工程化的直接证据。

### 6.3 Context Assembly 的来源（Figure 6）

| # | 源 | 时机 | 注入位置 |
|---|---|---|---|
| 1 | system prompt（含 output style / `--append-system-prompt`） | startup | system |
| 2 | env info（git status，`BREAK_CACHE_COMMAND` 注入） | startup（memoized） | system |
| 3 | CLAUDE.md 4 级层级 | startup（memoized） | **user message**（不是 system！） |
| 4 | path-scoped rules（`.claude/rules/*`） | lazy on dir read | user |
| 5 | auto memory | startup | user |
| 6 | tool metadata（含 ToolSearch 延迟加载） | per-turn / on demand | tool surface |
| 7 | conversation history | carry forward | user/assistant |
| 8 | tool results（file/cmd/subagent summary） | runtime | user |
| 9 | compact summaries | 替换老段 | user |

**最反直觉**：**CLAUDE.md 不是 system prompt**，是 user-context message —— 所以模型遵守它是**概率性**而非**确定性**的。**确定性**那一半交给 deny-first 权限规则。

### 6.4 Beyond compaction，4 个 context 节流配套

- **CLAUDE.md 嵌套懒加载**（仅当 read 该目录文件时）
- **Deferred tool schemas**（ToolSearch 开时只先挂工具名）
- **Subagent summary-only return**（subagent 全 transcript 落 sidechain，**parent 只看 summary**）
- **Per-tool-result budget**（防一条 verbose 输出独吞 context）

### 6.5 Reasoning vs Enforcement 分离

社区估计 codebase 中 **AI 决策逻辑仅 ~1.6%，operational infrastructure 占 ~98.4%**——验证了 minimal scaffolding 原则。

### 6.6 7-layer 安全栈

```
Tool pre-filter → Deny-first rules → Mode constraints → Auto-mode classifier
→ Shell sandbox → Resume 不恢复权限 → Hook 拦截
```

**任意一层都能拒**。其中 "resume/fork 不恢复 session-scoped 权限" 是个**反直觉但正确**的安全设计——session 是隔离的信任域。

### 6.7 Orchestration 选择

- 主循环：**ReAct reactive loop**（Yao 2022），不做 backtracking、不做 search tree
- 委托：**orchestrator-workers**（Schluntz & Zhang 2024）—— Agent tool dispatch subagent，重新进 `queryLoop()`，独立 context window，回 summary

## 7. 限制

**作者承认**：

- v2.1.88 是 snapshot，feature flag 行为可能随版本漂移
- "Long-term human capability preservation" 只作为 evaluative lens 不作为 co-equal value——与 Anthropic 自己的 132 人调研里"paradox of supervision"形成张力（Shen & Tamkin 2026 报告 AI-assisted 条件下 comprehension 测试低 17%）
- 部分推断依赖社区分析与官方文档，源码注释不一定权威

**我看出的**：

- 没量化每层 compaction 的实际触发率与节省 token 比例（除了 0.76% cache_creation 那一处）；外部读者无法估"如果我去掉某一层会怎样"
- compaction 的"用户不可见性"作者点了一句但没展开评估——**collapse 完全无 UI、microcompact 只一个 marker**——对 debugging 和审计是个真问题
- 5 层管线**复杂度成本**（feature flag 组合爆炸）作者承认但没给治理方案
- 与 OpenClaw 的对比只是结构性的，**没量化基准**——读者拿不到"哪种范式在哪种任务上 token/latency 更优"
- 全文围绕 v2.1.88 单点截面，对**架构演化**讨论极少（哪些层是逐版本叠出来的、哪些是 day-1 设计）

## 8. 与我关心的问题的联系（context assemble / compact / cache）

这是这篇我**最该带走的东西**，按工程槽位逐条对应。

### 8.1 Context Assembly 的工程范式

把"context 怎么组"显式分成 6 个轨道（system / project config / memory / conversation / runtime / on-demand），每个轨道**有不同的 mutability 与 load timing**。给 managed-agent 系统的直接启发：

- **memoize 任何能 memoize 的**（git status、CLAUDE.md hierarchy 都是 session-level 一次性算）
- **system prompt vs user message 的位置选择是个真决策**——CLAUDE.md 故意放 user message 是为了"让模型有判断空间，确定性留给权限层"
- **lazy-load by path**（嵌套 CLAUDE.md / path-scoped rules 只在读该目录文件时载入）这是省 context 最大的杠杆之一

### 8.2 Compaction Pipeline 的工程范式（最重要）

> **抄作业要点**：不要做"单层 compaction"，做"**渐进式 + cache-aware + read-time projection** 三件套"。

- **渐进式**：便宜的先（限单条输出大小 → 删老 → 局部摘要），贵的（全文 LLM 摘要）最后
- **cache-aware**：compaction 决定要等 API 真返回 `cache_deleted_input_tokens` 才下手，不要靠估值——这是把 prompt cache TTL/前缀语义当作一等公民的设计
- **read-time projection**：不要修改原 history，让"压缩"变成视图——这样 resume/fork/rewind/审计都成立。**这是 append-only 与 compaction 共存的关键工程发明**

我们做 managed agent 时，应当：
1. 先做 budget（单条工具结果限长） + 一个 LLM 摘要兜底——再迭代加 cache-aware 与 collapse
2. **永远** 让 compact summary 与原 history 并存，至少在磁盘上
3. 把 compaction boundary 序列化进 transcript（用 headUuid/anchorUuid/tailUuid 这种 chain patch 方案），而不是简单删除

### 8.3 Cache 协同的工程要点

- prompt cache **5 分钟 TTL**（论文 11.7 节里 KAIROS / SleepTool 部分明确提到）—— 任何"主动 sleep"都意味着重新 cache_creation
- microcompact 的 **CACHED_MICROCOMPACT** 路径表明：cache invalidation 应该是 compaction 决策的一等输入
- 那条 `0.76% of fleet cache_creation` 注释 = **舰队级证据**：cache hit/miss 是真实钱
- 工程取舍：**compaction 操作如果会破坏前缀，就值得为此推迟到 API 真返回再决定**

### 8.4 ComMer 与本文的对照（互补不冲突）

| | ComMer | Claude Code |
|---|---|---|
| 路线 | latent compaction（soft prompt + mean pool） | textual compaction（多层渐进 + summary） |
| 适用 | 风格类记忆（个性化偏好） | 任意对话历史 + 工具结果 |
| 不适用 | 事实记忆（destructive interference） | （任何要"高保真细节"的回顾——auto-compact 是有损的） |
| 对 cache 友好性 | 与 cache 无关（一组 latent vector） | 强依赖 prompt cache，cache-aware 是核心 |

**结论**：managed agent 的 memory 系统**不是二选一**——
- 用户偏好/口吻 → ComMer 路线（latent）
- 工具结果 / 错误日志 / 可检索事实 → RAG / structured note
- 长对话历史 → Claude Code 风格的 5 层 textual compaction + read-time projection

### 8.5 不该套的场景（红线）

- **审计/合规要求高的 agent**：context collapse 完全无 UI、microcompact 只一个 marker——监管场景里这种"用户不可见的压缩"是阻断式问题，必须改成"compaction 全部产物显式落盘 + 可重放"
- **多租户共享 gateway**：Claude Code 的 deny-first per-action + 单进程模型不适合 perimeter 安全场景，参考 OpenClaw 的 gateway 范式
- **小 context 模型**：5 层管线在 200K/1M 里是"省 token"，搬到 8K-32K 模型上要重新算便宜/贵的次序——budget 与 snip 可能直接顶到天花板，collapse 与 auto-compact 收益反而被放大

### 8.6 立刻能用的代码地标清单

研究 Claude Code 实现时直接跳这几个文件：
- `query.ts:365–453` — compaction pipeline 五连发
- `compact.ts` — `compactConversation()`、`buildPostCompactMessages()`、`annotateBoundaryWithPreservedSegment()`
- `context.ts` — `getSystemContext()` / `getUserContext()` 的 memoize 模式
- `claudemd.ts` — 4 级 memory + `@include`
- `sessionStorage.ts:247` — sidechain transcript 写法
- `runAgent.ts` / `AgentTool.tsx` — subagent 重入 `queryLoop()` 但 context 隔离
