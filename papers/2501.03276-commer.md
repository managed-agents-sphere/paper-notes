# ComMer: Compress and Merge

## 1. TL;DR

ComMer 把每篇用户文档独立压成一组 soft prompt（用冻结 LLM + LoRA），再用 **mean pooling** 合并成一个固定大小的潜变量喂回冻结 LLM。**风格类**个性化任务（改写、标题生成）越压越好且呈幂律；**知识密集**任务（QA）越压越差，因为均值池化会通过 destructive interference 抹掉文档细节。**真正的贡献不是 SOTA，而是这条边界**。

### 说人话

> ComMer 让一个**写过很多东西的用户**把这些资料"打包成一小撮数字"喂给 LLM，让 LLM 模仿他的风格说话；但**不能用这招让 LLM 记住他的具体事实**。

#### 术语翻译

| 原词 | 说人话 |
|---|---|
| **soft prompt** | 不是给 LLM 看一段文字，而是直接塞给它一组**数字向量**——像"机器内部速记"，比文字紧凑得多 |
| **冻结 LLM + LoRA** | 主模型权重不动（冻结），旁边挂一个**小到几 MB 的"适配器"**（LoRA）只训这小块。便宜、不破坏原模型 |
| **mean pooling** | 把 N 篇文档各自压出来的向量**按位求平均**。结果是：3 篇也好、800 篇也好，**最后只占同样大小的"位子"** |
| **潜变量 / latent** | 模型内部的隐藏状态（数字），不是文字。LLM 自己看得懂，人看不懂 |
| **幂律** | 文档数翻倍 → 质量按固定比例提升。log 坐标里是直线，**没饱和**，加越多越好 |
| **destructive interference** | 借自物理：两个方向相反的波叠加会互相抵消。这里指"两份文档的细节信号互相抵消、变成 0" |
| **SOTA** | state of the art，最优记录 |

#### 为什么"风格能压、事实不能压"

**风格类（求交集）**
- 你过去 100 条 tweet：都偏短、爱用 emoji、有点自嘲
- 平均 100 个向量 → "短 + emoji + 自嘲"这部分**重叠的信号被加强**，每条独有话题被稀释
- 这恰好就是你想要的——"用户风格" = 所有内容的**公共部分**

**事实类（求并集）**
- 笔记本：第 1 篇"我老婆叫张三"、第 2 篇"去年跑步 200 公里"、第 3 篇"会议室在 5 楼"
- 平均这三个向量 → 一坨**没人能解读**的混合体
- 你想要的"用户知识" = 每篇文档的**独有部分**，恰恰是平均会抹掉的

#### "这条边界才是真正贡献"什么意思

ComMer 的实验数字本身没破纪录。但它**第一次把"哪些记忆能压、哪些不能压"切得清清楚楚**——对做 managed agent 有用：以后设计 agent 长期记忆，**用户偏好/口吻**走压缩路线，**事实记忆**老老实实走 RAG，别瞎套。

## 3. 问题与动机

LLM 个性化只有两条路，都不够好（Figure 1）：

| 路径 | 优点 | 缺点 |
|---|---|---|
| **Prompt 工程 / RAG** | 不训练、响应快 | context window 限制；token 成本随用户文档线性增长；positional bias / lost-in-the-middle |
| **Fine-tuning / PEFT** | 推理 token-frugal | 每用户单训；新数据要重训；少数据用户质量差 |

ComMer 想拿"中间地带"：**训练成本一次性摊到所有用户、推理 token 极少、新文档来时 O(1) 增量更新**（只压新的、再融进现有均值）。

## 4. 方法

三步流水（Figure 2）：

### 4.1 Compression（每篇文档独立压）
- 文档后拼接 *k* 个**可训练 compression embeddings**（*k* ∈ {4, 8, 16, 32, 64, 128}）
- 整段过**冻结 Gemma-2b** + 一个**可训练 LoRA**（rank 64, α 16, dropout 0.1）
- 取**最后一层在那 *k* 个 compression embedding 位置上的隐藏向量**作为该文档的软提示

### 4.2 Merging（合并）
对 *n* 篇文档的 *k×d* 软提示**沿文档维 mean pooling** → 得到一个 *k×d* 的聚合软提示。三个性质至关重要：
- **形状与文档数无关**（推理预算固定）
- **顺序无关**（消除 positional bias）
- **可预计算 + O(1) 增量更新**

### 4.3 Response
聚合软提示与 query `x` 的 token embedding 拼接，喂给同一份**冻结 Gemma-2b**（生成器与压缩器共享 backbone）。

### 4.4 训练
- 监督形式：`⟨{doc_j}, x_i, y_i⟩`，对 `y_i` 做 cross-entropy
- 梯度只回传到 **LoRA + compression embeddings**，backbone 始终冻结
- 学习率不均匀关键：LoRA `1e-4`、compression embedding `1e-2`、prompt-tuning baseline `3e-2`
- 多文档 auto-encoding 预训练（FineWeb，1M 例，1 epoch）：把第 i 篇前缀 "Document i:"，问 "What does document n contain?"，要求重构原文

## 5. 实验

- **Backbone**：Gemma-2b-it（生成器 = 压缩器，共享权重）
- **数据集**：
  - **风格类**：LaMP-tweet paraphrasing、LaMP-news headline generation（BM25 取最多 8 篇历史）
  - **知识类**：PerLTQA（合成长期个人记忆 QA，27 personas 训练 / 5 测试）
- **Baseline**：**prompt-tuning**（Lester et al. 2021）—— 把 *k* 个可训练 soft prompt 接在所有原始文档拼接前面
- **指标**：**perplexity**（主）+ ROUGE-L（噪声大，仅辅助）

## 6. 关键结果

### 6.1 风格类：ComMer 完胜（在 token 预算受限时）
- ≤128 tokens 预算内，ComMer 用更少 token 得到更低 perplexity（Figure 3）
- **perplexity 随文档数呈幂律**（Figure 4）—— 暗示更多文档可继续涨

### 6.2 知识密集：ComMer 失败（关键负面结论）
- prompt-tuning **对噪声文档鲁棒**
- ComMer **越多文档越差**（Figure 5）—— mean pool 通过 destructive interference 抹掉细节
- **1 篇文档时最好**

### 6.3 预训练消融（Table 1）
| | 无预训练 | FineWeb | tweets | FineWeb+tweets |
|---|---|---|---|---|
| 平均 perplexity ↓ | 2.144 | 2.127 | **2.124** | 2.123 |

**有/无差距显著，但预训练数据选择影响很小。**

### 6.4 文档数量泛化（Figure 6）
- 测试时塞**更多**文档 → 更好（即便只用 1 篇训练）
- 测试时**少于**训练时 → 退化
- 最反直觉：**只用 1 篇训练（从未与 mean-pool 联训）**，测试时多篇平均仍提升 → 印证 *linear representation hypothesis*（Park 2024、Bricken 2023、Elhage 2022）

### 6.5 替代合并策略（Figure 7）
- **Concat 压缩结果**：质量反而比 mean pool 差（注意力被分散）
- **先 concat 文档再压一次**：略好于 vanilla，但每来新文档要重压所有历史 → 不划算
- 结论：vanilla mean-pool 是 Pareto 最优；且**只有 mean-pool 能配合 retrieval**（concat 思路无法预计算）

## 7. 限制

**作者承认**：
- 不适合事实型上下文，是结构性限制
- 数据集文档数偏少（≤8），幂律外推到长程记忆未经验证
- 没和 PPlug / OPPU / HYDRA 直接 head-to-head（只在相关工作里做了概念对照）
- 隐私/偏见未处理（Impact Statement 直说"本文不解决"）

**我看出的**：
- 只在 Gemma-2b 上做。soft prompt 路线在更大模型上是否仍占优是开放问题
- 共享 backbone（生成器 = 压缩器）是个工程取巧，扩到不同 backbone 时的成本未量化
- 评测严重依赖 perplexity，作者自己也承认 ROUGE-L 噪声大——下游真实任务效果存疑

## 8. 与我关心的问题的联系（managed agent / context engineering）

这篇的真正价值是**划清边界**，对设计 agent memory 系统非常关键：

1. **Memory hierarchy 设计的实证依据**
   Anthropic *Context Engineering for Agents* 在讨论 long-horizon context 时引这篇，正是因为它把"什么记忆能压"答得清楚：
   - **用户偏好 / 行为风格 / tone / habits** → 适合 latent compaction（ComMer 路线）
   - **可检索事实 / 工具调用结果 / 错误日志** → 必须 RAG / structured note / 完整 token 保留

2. **Position-invariant + O(1) 增量**
   Agent 长期记忆系统在工程上必须有"加新条目不重算全部历史"的能力。mean pool 是最朴素也最实用的实现。

3. **Linear representation hypothesis 的工程证据**
   Figure 6 那个"只用 1 篇训练、测试时多篇平均仍提升"的发现——说明只要有一个好的单文档压缩器，**多文档合并几乎不需要再训**。这对 agent 工程意义很大：可以单独优化压缩器，再搭组合策略。

4. **训练成本摊销 = managed agent 的天然契合点**
   一次训练、所有用户共享，符合 managed agent 不为每用户单独 fine-tune 的工程哲学。

**反过来看，要警惕**：
- 如果 agent 任务里事实保真度重要（工具结果、API 响应、错误堆栈），**不要** ComMer 化，老老实实 token 保留或 RAG
- 它的 perplexity 优势在 ≤128 token 预算下才显著，更宽预算下与 prompt-tuning 接近
